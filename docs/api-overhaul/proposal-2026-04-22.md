# API Category Restructure Proposal

Status: proposed
Date: 2026-04-23
Related: #932, #904, #868

## Context

`#904` asks for a consumer-first API overhaul: clearer names, staged discovery in `/api/docs`, tighter path semantics, and promotion of high-frequency skill workflows into first-class API surfaces. `#932` is the design-only Phase 1 for that work. No endpoint changes happen in this phase; the goal is to lock the target taxonomy, migration contract, and breakage surface before the post-PG cleanup implementation.

The current surface already has some of the physical routing groundwork in `src/server/routes/domains/`, but that structure is still server-internal. Agents and dashboard clients still see a flat `/api/docs/{category}` model plus several legacy prefixes (`/api/re-review`, `/api/send_to_agent`, `/api/github-issues`, `/api/hook/session`, `/api/auto-queue/activate`) that do not present one coherent mental model.

## Live Snapshot (2026-04-23 KST)

Snapshot sources:

- live `GET /api/docs`
- live `GET /api/docs?format=flat`
- generated inventory in `docs/generated/route-inventory.md`

Observed state:

- `/api/docs` currently returns `29` categories, not `28`
- `/api/docs?format=flat` currently returns `180` documented endpoints
- `docs/generated/route-inventory.md` currently lists `189` HTTP routes
- the docs inventory and the route inventory are already out of sync

This proposal treats the live API as canonical and makes the current drift explicit instead of preserving stale counts from the issue text.

### Inventory Mismatch That Should Drive The Redesign

The current docs surface is not only flat; it is also incomplete. As of the snapshot above, these routes exist in the route inventory but are missing from `/api/docs`:

- `DELETE /api/hook/session/{sessionKey}`
- `GET /api/round-table-meetings/channels`
- `GET /ws`
- `POST /api/auto-queue/runs/{id}/entries`
- `POST /api/auto-queue/runs/{id}/restore`
- `POST /api/auto-queue/slots/{agent_id}/{slot_index}/rebind`
- `POST /api/hook/reset-status`
- `POST /api/hook/skill-usage`
- `POST /api/inflight/rebind`
- `POST /api/internal/escalation/emit`

There is also one reverse drift item: `/api/docs` advertises `GET /api/stats/memento`, while the generated route inventory does not list it. Phase 2 should not just rename routes; it should make `/api/docs` generation authoritative over the real router.

## Proposal Summary

The target model is:

1. `GET /api/docs` returns `7` external top-level groups.
2. `GET /api/docs/{group}` returns subcategories inside that group.
3. `GET /api/docs/{group}/{category}` returns endpoint details.
4. `internal` is preserved as a hidden namespace, but it is not listed in the root response.

The hidden-`internal` rule resolves the current issue/epic inconsistency cleanly:

- user-facing discovery stays at `7` groups
- internal runtime/debug endpoints still have a stable documentation address
- agent consumers are not nudged toward runtime-only helpers

### Proposed External Groups

| Group | Purpose |
| --- | --- |
| `runtime` | Health, auth, live session state, message log, and API discovery surfaces |
| `kanban` | Cards, repos, pipeline, review, and PM decision workflow |
| `agents` | Agent registry, office/department topology, and skill metadata |
| `integrations` | Discord, GitHub, and round-table meeting workflows |
| `automation` | Dispatches, queue orchestration, auto-queue, and cron-style execution helpers |
| `config` | Settings, onboarding, and policy/configuration surfaces |
| `observability` | Analytics, stats, receipts, and operator-facing diagnostics |

Hidden group:

| Group | Purpose |
| --- | --- |
| `internal` | Runtime-owned endpoints such as thread reuse and operator-only escape hatches |

## Endpoint Namespace Cleanup Map

The grouped docs structure only works if the runtime paths themselves stop advertising conflicting concepts. The table below lists the concrete path cleanup that Phase 2 should implement.

| Current path or prefix | Proposed path or prefix | Why |
| --- | --- | --- |
| `/api/docs/{category}` | `/api/docs/{group}/{category}` | Introduce staged discovery instead of one flat level |
| `/api/re-review` | `/api/kanban-cards/batch-rereview` | Remove orphan prefix; keep rereview inside the card namespace |
| `/api/kanban-cards/bulk-action` | fold into `/api/kanban-cards/batch-transition` | One batch transition endpoint is easier to document and reason about |
| `/api/kanban-cards/{id}/force-transition` | `/api/kanban-cards/{id}/transition` | Force becomes an option on one transition contract, not a separate noun |
| `/api/auto-queue/activate` | `/api/queue/dispatch-next` | Name the side effect directly |
| `/api/auto-queue/dispatch` | removed | `generate` and `dispatch-next` already cover creation vs execution |
| `/api/auto-queue/*` and `/api/channels/{id}/queue` split | `/api/queue/*` | Queue APIs should share one namespace |
| `/api/hook/session` | `/api/dispatched-sessions/webhook` | Session hook belongs to dispatched-session lifecycle |
| `/api/send` | `/api/discord/send` | Normalize Discord under one prefix |
| `/api/send_to_agent` | `/api/discord/send-to-agent` | Same reason; remove mixed snake/camel naming |
| `/api/senddm` | `/api/discord/send-dm` | Same reason |
| `/api/issues` | `/api/github/issues/create` | Skill promotion target should live under the GitHub namespace |
| `/api/github-issues*` | `/api/github/issues*` | Collapse dashboard read model and integration write model |
| `/api/github-repos` | `/api/github/repos*` | Stop maintaining a parallel top-level prefix for one consumer |
| `/api/github-closed-today` | `/api/github/issues/closed-today` | Keep derived GitHub read models inside the GitHub namespace |
| `/api/kanban-reviews/{id}/decisions` | `/api/reviews/{id}/decisions` | One reviews prefix |
| `/api/kanban-reviews/{id}/trigger-rework` | `/api/reviews/{id}/trigger-rework` | Same reason |
| `/api/review-verdict` | `/api/reviews/verdict` | Same reason |
| `/api/review-decision` | `/api/reviews/decision` | Same reason |
| `/api/review-tuning/aggregate` | `/api/reviews/tuning/aggregate` | Same reason |

## Skill Promotion Candidates

### `memory`

`memory-read` and `memory-write` are frequent, first-order behaviors for agents rather than niche operator workflows. Keeping them as shell-oriented skills forces prompt text, docs examples, and client integrations to describe an indirection layer instead of the capability itself. That is the wrong abstraction boundary for a system that already teaches agents to inspect `/api/docs` before guessing.

Promoting memory into `/api/memory/*` makes the capability discoverable in the same place as the rest of the platform. It also gives the runtime one contract for backend selection and observability instead of scattering the behavior across skill bodies, prompt conventions, and optional local file layouts.

### `github`

Issue creation is currently split between a skill wrapper and multiple GitHub API surfaces. That is acceptable for local operator convenience, but it is weak as a platform contract because naming, repo allowlists, and template behavior are not anchored in one API path. The current `POST /api/issues` path is also too generic for a mixed integration surface.

Promoting issue creation into `/api/github/issues/create` aligns the path with the real domain, makes the skill body optional instead of required, and gives the dashboard, Discord-side agents, and external scripts one place to target when they need a GitHub write operation.

### `sessions`

`watch-agent-turn` is not really a skill from the consumer point of view. It is a runtime inspection API disguised as a skill because there is no stable endpoint for "show me what this agent session is doing right now." That makes dashboard features and external recovery tooling depend on shell wrappers or tmux-specific glue.

Promoting session inspection into `/api/sessions/{session_id}/tmux-output` turns that behavior into a real, documented runtime surface. It also creates a clean home for other session-centric read models that are currently split between `agents`, `sessions`, and `dispatched-sessions`.

## Dual-Mode Memory Interface

The memory API should expose one stable contract and make backend choice explicit. The target is not full Memento feature parity on day one; the target is one consumer-facing contract that can route to the right implementation.

### Proposed Endpoints

- `POST /api/memory/recall`
- `POST /api/memory/remember`
- `POST /api/memory/forget`
- optional follow-up: `GET /api/memory/capabilities`

### Backend Modes

| Mode | Meaning |
| --- | --- |
| `auto` | Prefer Memento when configured, otherwise fall back to local file-backed storage |
| `memento` | Require Memento; fail loudly if unavailable |
| `file` | Require local file-backed storage; useful for development and offline recovery |

### Contract Rules

1. The request schema stays stable across backends.
2. Every response includes `backend: "memento" | "file"` so callers know what actually served the request.
3. `mode=auto` may fall back; explicit modes must never fall back silently.
4. The API returns capability gaps as structured errors rather than pretending both backends are identical.
5. The docs examples should focus on the common subset first: query, topic, keywords, scope, workspace, and write/delete outcomes.

### Suggested Envelope Shape

```json
{
  "backend": "memento",
  "mode": "auto",
  "result": {
    "items": []
  },
  "warnings": []
}
```

This avoids leaking backend-specific details into every caller while still letting advanced clients branch on the real execution mode.

## Migration Strategy

### Phasing

| Phase | Scope |
| --- | --- |
| `#932` | Design-only proposal and migration contract |
| `#904-P2` | Docs hierarchy + endpoint/path cleanup after `#868` |
| `#904-P3` | Skill promotion implementation (`memory`, `github`, `sessions`) |
| `#904-P4` | Dual-mode memory backend implementation and backend-specific tests |

Phase 2 starts only after user confirmation on this proposal. This document does not authorize path changes by itself.

### Alias And Deprecation Policy

This proposal recommends:

- runtime alias period: `0 days`
- docs migration notice period: `1 release cycle`
- implementation PR requirement: old and new paths must not coexist in release after the cutover commit lands

Rationale:

- the parent epic already assumes a breaking change posture
- keeping both old and new route trees live would double the API docs surface and prolong ambiguity
- the real safety lever is not long-lived aliases; it is updating dashboard/CLI/scripts/prompts in the same PR and shipping a precise migration appendix

### Rollout Sequence

1. Merge this proposal.
2. Wait for `#868` to complete.
3. Build the Phase 2 implementation as one intentional migration PR:
   - docs hierarchy
   - path cleanup
   - dashboard/CLI/script/prompt updates in the same branch
4. During development and review, allow docs to annotate replacements.
5. On release cutover, remove the legacy paths immediately and keep only the written migration table.

### Required Deprecation Material

The Phase 2 PR description should include:

- old path -> new path table
- changed request field names
- removed endpoints with replacement or "no replacement"
- dashboard/CLI/script migration checklist

## Expected Breaking Changes And Affected Clients

The main risk is not server code churn by itself. The main risk is that many first-party consumers already hardcode the old paths.

| Consumer | Evidence in repo | Expected impact |
| --- | --- | --- |
| Dashboard | `dashboard/src/api/client.ts` uses `/api/kanban-reviews/*`, `/api/github-issues*`, `/api/auto-queue/activate` | Must migrate review, GitHub, and queue clients in lockstep with server change |
| Rust CLI | `src/cli/client.rs` uses `/api/auto-queue/dispatch` and many legacy kanban/runtime routes | CLI commands will break immediately unless the same PR updates request paths |
| Discord runtime internal client | `src/services/discord/internal_api.rs` uses `/api/hook/session` | Session hook rename is a runtime-critical break, not a docs-only break |
| Shell automation | `scripts/queue-stability-batch.sh` uses `/api/auto-queue/activate` | Existing operational scripts will fail silently or half-complete if not migrated |
| Prompt and training text | `src/services/discord/prompt_builder.rs`, `src/cli/init.rs`, `config/agents/_shared.prompt.md` reference `/api/docs` discovery patterns | Docs examples and path names must stay synchronized with the new hierarchy |
| External curl users | not greppable in repo | Any user or notebook that copied current `/api/docs/{category}` or legacy prefixes will need the migration table |

### High-Risk Breaking Surfaces

- queue execution naming: `activate` vs `dispatch` vs `dispatch-next`
- GitHub path split: `/api/issues` + `/api/github-issues*` + `/api/github/repos*`
- reviews split: `/api/kanban-reviews/*` + `/api/review-*`
- session hook rename: `/api/hook/session`
- Discord delivery split: `/api/send`, `/api/send_to_agent`, `/api/senddm`
- docs navigation shape: flat `/api/docs/{category}` to grouped `/api/docs/{group}/{category}`

## Recommendation

Proceed with the redesign, but keep three constraints explicit in the implementation phase:

1. Treat the live API inventory, not the ticket text, as the source of truth. The current surface is `29` categories and `180` documented endpoints.
2. Keep `/api/docs` user-facing at `7` groups and move `internal` behind an explicit opt-in path.
3. Ship path cleanup only with first-party client migration in the same PR. Alias-free rollout is viable only if dashboard, CLI, Discord runtime, scripts, and prompt docs move together.
