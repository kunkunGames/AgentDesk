# API Overhaul Proposal

Date: 2026-04-22  
Source of truth checked: live `GET /api/docs` on 2026-04-23 KST

## 1. Scope and baseline

This proposal is a docs-only restructuring plan for the public API surface. It does not rename handlers or change server behavior in this change set. The immediate goal is to replace the current flat `/api/docs/{category}` map with a grouped information architecture that is easier for humans and clients to navigate.

### Inventory observations

- `internal` should not be treated as ordinary public documentation.

## 3. Proposed seven-group documentation model

The docs tree should move from flat categories to seven stable groups:

| Group | Purpose |
| --- | --- |
| `runtime` | live execution, dispatching, sessions, messaging, auth, docs, health |
| `kanban` | cards, reviews, pipeline, PM decisions, repo board config |
| `agents` | agent registry, offices, departments, assignment topology |
| `integrations` | Discord and GitHub entry points |
| `automation` | queue automation, scheduling, meeting orchestration |
| `config` | onboarding, settings, policies, skill catalog exposure |
| `observability` | analytics, stats, audit-style operational insight |

`internal` should remain documented only behind an explicit internal marker, not promoted into the main public group navigation.

## 5. Skill promotion candidates

Three areas are mature enough to be documented as promoted first-class API capabilities instead of remaining implicit implementation details.

### `memory`

Repo-local skills already expose a stable conceptual surface:

- `memory-read`
- `memory-write`
- `memory-merge`

These should be promoted in docs as a coherent memory capability family rather than described only as skill internals. The public docs should explain read, write, and maintenance operations in one place even if the transport remains skill-driven.

### `github`

GitHub already exists as a hybrid of repo registration, issue creation, sync, dashboard queries, and close actions. The docs should promote this from two fragmented categories into one integration narrative with repo management and issue operations under one heading.

### `sessions`

`sessions` and `dispatched-sessions` currently read like separate technical implementation buckets. They should be promoted as a single runtime session story:

- session listing and termination
- dispatched-session lifecycle
- webhook or hook-style thread linkage
- stale-session cleanup

## 6. Dual-mode memory interface

The skill definitions for `memory-read` and `memory-write` already imply a backend-agnostic contract with `backend: auto | memento | file`, where `auto` resolves `memento -> file`. That should become an explicit docs principle.

Recommended contract:

- present one memory interface to clients
- document backend selection as an execution detail, not a separate product surface
- keep request semantics stable whether data resolves through memento or file-backed memory
- reserve backend-specific diagnostics for advanced troubleshooting sections

This dual-mode model matters because it prevents API consumers from binding to the current storage mechanism. Clients ask for memory operations; the server decides whether memento or file-backed storage satisfies the request.

## 7. Migration, deprecation, and alias period

The overhaul should be staged in three phases.

### Phase 1: grouped docs launch

- add the seven-group docs tree
- preserve all current `/api/docs/{category}` pages
- annotate each old page with its new grouped location

### Phase 2: endpoint alias period

- introduce preferred endpoint spellings only where naming is materially inconsistent
- keep legacy routes active
- mark legacy routes as deprecated in docs, examples, and generated references

### Phase 3: removal window

- remove legacy docs pages only after one full release cycle with aliases in place
- remove legacy endpoints only after client telemetry confirms low usage
- keep `internal` excluded from general client-facing migration promises

A reasonable policy is one release cycle for docs aliases and at least one additional release cycle for endpoint aliases.

## 8. Breaking change and client impact

The docs regrouping itself should be non-breaking if old docs URLs continue to resolve. The real risk appears when path cleanup turns into handler-level endpoint changes.

Expected client impact:

- hard-coded docs links will break unless category aliases remain in place
- generated SDKs or custom scripts may depend on current path spellings such as `/api/send`, `/api/issues`, or `/api/github-issues`
- dashboard consumers may treat `github` and `github-dashboard` as separate products and need a merged mental model
- internal operators may rely on flat category naming when triaging runtime incidents

The safest rule is:

- docs IA changes may ship immediately with aliases
- transport path changes should not ship without alias coverage, changelog callouts, and usage validation

## 9. Recommendation

Adopt the seven-group docs tree first, publish the remap table as the canonical migration guide, and treat `memory`, `github`, and `sessions` as promoted capability families in the new information architecture. This delivers a cleaner public API narrative without coupling documentation progress to immediate handler refactors.
