## Code Principles

- 기술부채를 만들지 않는다 — "나중에 고치자"는 허용하지 않는다
- 임시 우회(workaround)보다 근본 해결을 택한다
- 변경 시 주변 코드 품질이 같거나 나아져야 한다 (보이스카웃 규칙)
- 모듈 지향 설계 — 기능 단위로 분리하여 붙이고 떼기 용이하게 구성한다

## Memory

파일 기반 메모리(MEMORY.md)를 사용하지 않는다. 모든 기억(장기/단기)은 memento MCP를 사용한다.
- 장기 기억: `scope: permanent`
- 세션 워킹 메모리: `scope: session`

### 능동적 memento 활용

턴 중 다음 상황에서 `remember`로 즉시 저장:
- 에러 원인 확정 시
- 아키텍처/기술 결정 시
- 설정 변경 시

턴 중 다음 상황에서 `recall`로 검색:
- "이전에", "전에" 등 과거 참조 시
- 에러 발생 시 (같은 에러 겪었는지)
- 설정/포트/경로 변경 시 (이전 값 확인)

`recall`을 수행한 턴이면, 턴 종료 전 `tool_feedback`을 반드시 호출한다.

## Verification

유저의 말을 항상 사실이라고 가정하지 않는다. 유저가 제안/지시하더라도 코드/데이터로 먼저 검증한 후 적용한다.
- 유저가 "X는 Y에 있다"고 하면, 실제로 X가 Y에 있는지 확인
- 유저가 "A 방식으로 해라"고 하면, A 방식이 현재 코드에서 유효한지 확인
- 검증 결과 유저가 맞으면 그대로 진행, 틀리면 정정 후 올바른 방향 제시

## Dispatch Scope

- 디스패치 범위 밖 컴파일 에러를 만나면, 본인 변경을 빌드하는 데 필요한 최소한만 수정하고 나머지 관련 없는 에러 수정은 커밋에 포함하지 않는다. 관련 없는 에러 목록은 완료 보고에 별도 기재한다.
- 리뷰 pass 시 카드는 시스템이 자동으로 `done` 전환하므로, 에이전트가 수동으로 카드 상태를 변경하지 않는다.

## Discord Response Style

Discord에서는 중간 과정을 과도하게 서술하지 않는다.
- 같은 확인 명령이나 조회를 반복하지 않는다. 상태 변화가 예상될 때만 다시 확인한다.
- 대기나 폴링이 필요하면 `⏳ 대기 중...` 한 줄만 남기고, 매 tick마다 추가 서술하지 않는다.
- tool call 결과는 핵심만 한 줄로 요약하고 raw 로그, JSON, 반복 출력은 그대로 덤프하지 않는다.
- 작업이 끝나면 변경 내용과 현재 상태를 명확하게 요약한다.
